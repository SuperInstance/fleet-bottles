### 3. Critical Bottleneck Found
**Root cause:** `_encode_tile()` in `NerveTopology` creates a new `np.random.RandomState()` **150 times per tick** (once per tile, and there are ~150 tiles from 4 fibers × ~35 rooms). That's 70% of the 187ms tick time.

| Fix | File | Technique | Impact |
|-----|------|-----------|--------|

## What We Learned

1. **The neural substrate (RoomGrid) is only 20% of the cost.** The routing layer — the "nervous system" — was 80% of the cost. This is backwards. The nervous system should be faster than the brain.
2. **Python `random.random()` in tight loops is deadly.** 800 calls per tick × Python function call overhead = death.
3. **Vectorization beats algorithmic cleverness.** `fire_fast()` is 60× faster not because of better math, but because it batches random checks through numpy.
4. **Caching beats recomputation.** `_encode_tile()` was spending 99% of its time creating `RandomState` objects, not doing actual encoding.
5. **The Rust kernel exists but is barely wired.** `forward_rust()` is only 3× faster than numpy. It should be 160× faster. The `.so` loads but something is suboptimal in the call path.

---

## Open Questions / Next Frontiers

1. **Fiber perceive optimization** — After routing fix, `_extract_features()` is now the dominant cost (~29ms/tick). Needs vectorization or Rust rewrite.
2. **Rust kernel tuning** — `jepa_forward_batch` in `nerve/src/lib.rs` claims 10K rooms in 2.35ms. Current Python call path achieves only 3× speedup. Need to investigate ctypes overhead vs actual Rust performance.
3. **CUDA path** — For 10K+ rooms on RTX 4050. `plato_forge/jepa_kernel.cu` is the target.
4. **Compiler self-hosting** — Eventually the agentic compiler should compile itself. Currently it's Python-only.
5. **Chapel multi-node** — For 100K+ rooms distributed across fleet nodes. Deferred until single-node capacity exceeded.

---

## The Agentic Compiler Thesis ( distilled )

> Start hardware-agnostic. Profile at runtime. Identify hot paths. Recompile them to the best available backend (Numba→CUDA→Rust). Learn from usage. Repeat.

The **killer app** is self-acceleration: the more the fleet operates, the faster it gets. The compiler becomes a permanent resident — the ecosystem's metabolism, constantly optimizing its own biochemistry.

**The muscle memory concept applied to code:** the more you use a path, the faster it gets.

| **Fast hash + feature cache** | `nerve/fiber.py` | `_hash_signal()`: `.tobytes()` instead of SHA-256 on `str(np.array)`. `_extract_features()`: signal-type-aware (numeric features for arrays, not string digit/alpha checks). Feature cache by hash. | Fiber perceive: ~25ms → ~0.3ms (36×) |

## What Did NOT Work

| Attempt | Why It Failed | Lesson |
|---------|---------------|--------|
| **LRU cache on `_route_key`** | String formatting `f"{a}→{b}"` is already ~100ns. `functools.lru_cache` lookup is ~500ns + eviction overhead. **Net +1ms.** | Don't cache what is already fast. |
| **Rust kernel for <500 rooms** | `forward_rust()` uses ctypes FFI: `np.ascontiguousarray()` × 7 + `data_as(POINTER)` × 7 + C call overhead. For small arrays, this dominates actual computation. **Rust was slower than numpy for <500 rooms.** | Need adaptive backend: numpy for small, Rust for large. |
| **numpy `random.randn()` per tick** | `np.random.randn(64)` in fiber perceive loop is expensive. Replaced with pre-allocated buffer + `np.random.randn(signal_dim).astype(float32)` once, then `[:]` slice for each fiber. | Batch allocations. |

## Key Decisions

1. **Adaptive backend selector** (not yet implemented but validated): numpy for n<500, Rust for n≥500, CUDA for n≥10K.
2. **Signal-type-aware extraction**: If signal is `np.ndarray`, extract `mean`, `std`, `min`, `max`, `nonzero`, `entropy`. If string, use `length`, `hash_prefix`, `contains_digits/alpha`. If dict, recurse. This is 10× faster than generic `str(signal)` + regex.
3. **Ring buffer beats deque for numpy**: `deque.append(np_array.copy())` forces Python-level iteration and per-element copy. A 3D numpy array with a head pointer is pure C.
4. **Pre-compute beats laziness**: The 1024-vector LUT costs ~1ms at init, saves ~6ms per tick forever. The `np.random.RandomState(seed)` pattern (lazy per-call) was the single biggest cost in the entire system.

## What We Learned (Round 2)

1. **The 80/20 rule is recursive.** First pass killed the big bottlenecks (routing 120ms, tile 30ms). Second pass killed the medium ones (fiber 25ms, grid 7ms). Each pass finds the new dominant cost.
2. **Python string formatting is not free in tight loops.** `_route_key` called 4,000× per 20 ticks = 200×/tick. At 1μs each, that's 0.2ms — small but measurable when targeting <10ms.
3. **Numpy array → Python string is pathological.** `str(np.random.randn(64))` produces a 500+ character string. `hashlib.sha256(string.encode())` on that is ~0.5ms. `.tobytes()` + `hash()` is ~0.001ms.
4. **Pre-allocation > lazy allocation > repeated allocation.** The three levels of memory efficiency. We moved the system from level 3 (worst) to level 1 (best) in the hot paths.
5. **The Rust kernel is a red herring at small scale.** For 100 rooms, ctypes overhead dominates. For 10K rooms, Rust's SIMD matmul would win. The compiler needs an adaptive threshold.

---

## Open Frontiers (Still On The Table)

1. **Fiber perceive elimination for COMPILED fibers** — If a fiber is COMPILED and the signal hash matches, skip `_extract_features()` entirely. Currently still called but result ignored. Could save another 0.3ms.
2. **Rust kernel actual tuning** — `jepa_forward_batch` in `nerve/src/lib.rs` claims 10K rooms in 2.35ms. The ctypes Python wrapper is the bottleneck, not the Rust code. Options: (a) batch multiple ticks in one FFI call, (b) use `pyo3` instead of ctypes, (c) keep weights in Rust memory, not Python.
3. **CUDA path** — `plato_forge/jepa_kernel.cu` exists but is not wired. For 10K+ rooms on RTX 4050, this is the next frontier.
4. **Compiler self-hosting** — The agentic compiler (`sunset/compiler.py`) is still Python-only. It should eventually compile its own hot paths.
5. **Topology tick() → Cython/Numba** — The orchestration layer itself (not just the sub-components) could be JIT-compiled. The 2ms overhead is mostly Python loop overhead.
